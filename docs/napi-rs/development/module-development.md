# NAPI-RS Module Development Guide

This guide covers how to develop, build, test, and deploy NAPI-RS modules in the Turbo Asset platform.

## 🏗️ Architecture Overview

### Module Structure

Each NAPI-RS module follows a consistent structure:

```
packages/[service-name]/
├── src/
│   └── lib.rs              # Main Rust implementation
├── Cargo.toml              # Rust dependencies and configuration
├── package.json            # Node.js package configuration
├── build.rs                # Build script
├── index.d.ts              # TypeScript type definitions
└── README.md               # Module-specific documentation
```

### Core Components

1. **Rust Core (`src/lib.rs`)**
   - Business logic implementation
   - NAPI-RS bindings and exports
   - Error handling and validation
   - Async operations with Tokio

2. **Build Configuration (`Cargo.toml`)**
   - Rust dependencies
   - NAPI-RS configuration
   - Target platform specifications

3. **Node.js Integration (`package.json`)**
   - NPM package metadata
   - Build scripts and commands
   - Development dependencies

4. **Type Definitions (`index.d.ts`)**
   - TypeScript interfaces
   - Auto-generated by NAPI-RS

### 5. Implement Service (src/lib.rs)

```rust
use chrono::{DateTime, Utc};
use napi::{bindgen_prelude::*, Result};
use napi_derive::napi;
use serde::{Deserialize, Serialize};
use std::collections::HashMap;
use uuid::Uuid;

/// Base entity structure following the universal data standard
#[napi(object)]
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct BaseEntity {
  pub id: String,
  pub organization_id: String,
  pub created_at: DateTime<Utc>,
  pub updated_at: DateTime<Utc>,
  pub created_by: String,
  pub updated_by: String,
  pub version: i32,
  pub is_active: bool,
  pub metadata: Option<HashMap<String, String>>,
}

/// Standard response structure
#[napi(object)]
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct StandardResponse<T> {
  pub success: bool,
  pub data: Option<T>,
  pub error: Option<ErrorResponse>,
  pub metadata: Option<ResponseMetadata>,
}

#[napi(object)]
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct ErrorResponse {
  pub code: String,
  pub message: String,
  pub details: Option<HashMap<String, String>>,
}

#[napi(object)]
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct ResponseMetadata {
  pub timestamp: DateTime<Utc>,
  #[napi(js_name = "requestId")]
  pub request_id: String,
  #[napi(js_name = "executionTime")]
  pub execution_time: i64,
  #[napi(js_name = "apiVersion")]
  pub api_version: String,
}

/// My new service implementation
#[napi]
pub struct MyNewService {
  // Service state and configuration
}

#[napi]
impl MyNewService {
  #[napi(constructor)]
  pub fn new() -> Self {
    Self {}
  }

  /// Initialize the service
  #[napi]
  pub async fn initialize(&self, config: HashMap<String, String>) -> Result<bool> {
    // Initialize service with configuration
    // Connect to database, setup connections, etc.
    Ok(true)
  }

  /// Health check for the service
  #[napi]
  pub fn health_check(&self) -> Result<String> {
    Ok("Service is healthy".to_string())
  }

  /// Get service information
  #[napi]
  pub fn get_service_info(&self) -> Result<HashMap<String, String>> {
    let mut info = HashMap::new();
    info.insert("name".to_string(), "my-new-service".to_string());
    info.insert("version".to_string(), "1.0.0".to_string());
    info.insert("description".to_string(), "My new service description".to_string());
    Ok(info)
  }

  // Add your custom service methods here
  #[napi]
  pub async fn custom_operation(&self, input: String) -> Result<String> {
    // Implement your business logic
    Ok(format!("Processed: {}", input))
  }
}

/// Initialize the module
#[napi]
pub fn init() -> Result<String> {
  Ok("MyNewService initialized successfully".to_string())
}
```

    expect(result).toBe('Processed: test input');

## 🏛️ Design Patterns

### 1. Service Pattern

Each module implements a service class with standard methods:

- `new()` - Constructor
- `initialize(config)` - Service initialization
- `health_check()` - Health monitoring
- `get_service_info()` - Service metadata

### 2. Error Handling Pattern

```rust
use thiserror::Error;

#[derive(Error, Debug)]
pub enum ServiceError {
    #[error("Database connection failed: {0}")]
    DatabaseError(String),
    #[error("Configuration error: {0}")]
    ConfigError(String),
    #[error("Operation failed: {0}")]
    OperationError(String),
}

impl From<ServiceError> for napi::Error {
    fn from(err: ServiceError) -> Self {
        napi::Error::new(napi::Status::GenericFailure, err.to_string())
    }
}
```

### 3. Configuration Pattern

```rust
#[derive(Debug, Clone)]
pub struct ServiceConfig {
    pub database_url: String,
    pub redis_url: Option<String>,
    pub log_level: String,
    pub max_connections: u32,
}

impl ServiceConfig {
    pub fn from_hashmap(config: HashMap<String, String>) -> Result<Self> {
        Ok(Self {
            database_url: config.get("database_url")
                .ok_or_else(|| ServiceError::ConfigError("database_url required".to_string()))?
                .clone(),
            redis_url: config.get("redis_url").cloned(),
            log_level: config.get("log_level").unwrap_or(&"info".to_string()).clone(),
            max_connections: config.get("max_connections")
                .and_then(|s| s.parse().ok())
                .unwrap_or(10),
        })
    }
}
```

### 4. Async Operations Pattern

```rust
use tokio::sync::RwLock;
use std::sync::Arc;

#[napi]
pub struct AsyncService {
    state: Arc<RwLock<ServiceState>>,
}

#[napi]
impl AsyncService {
    #[napi]
    pub async fn process_data(&self, data: Vec<String>) -> Result<Vec<String>> {
        let mut results = Vec::new();
        
        // Process data concurrently
        let tasks: Vec<_> = data.into_iter().map(|item| {
            tokio::spawn(async move {
                // Async processing
                format!("processed: {}", item)
            })
        }).collect();

        for task in tasks {
            results.push(task.await.map_err(|e| {
                napi::Error::new(napi::Status::GenericFailure, e.to_string())
            })?);
        }

        Ok(results)
    }
}
```

## 🧪 Testing Strategies

### Unit Testing

Test individual functions and methods:

```rust
#[cfg(test)]
mod tests {
    use super::*;

    #[tokio::test]
    async fn test_service_initialization() {
        let service = MyNewService::new();

            ("database_url".to_string(), "postgres://test".to_string()),
        ]);
        
        let result = service.initialize(config).await;
        assert!(result.is_ok());

    #[test]
    fn test_health_check() {
        let service = MyNewService::new();

        assert!(health.is_ok());
        assert_eq!(health.unwrap(), "Service is healthy");

}
```

## 🚀 Performance Optimization

### Memory Management

```rust
use std::sync::Arc;
use tokio::sync::RwLock;

// Use Arc for shared ownership
#[napi]
pub struct OptimizedService {
    // Shared state across async operations
    shared_data: Arc<RwLock<HashMap<String, String>>>,
}

#[napi]
impl OptimizedService {
    #[napi]
    pub async fn efficient_operation(&self, key: String) -> Result<Option<String>> {
        // Read lock for concurrent access
        let data = self.shared_data.read().await;
        Ok(data.get(&key).cloned())
    }
}
```

### Database Connection Pooling

```rust
use sqlx::{PgPool, postgres::PgPoolOptions};

#[napi]
pub struct DatabaseService {
    pool: Option<PgPool>,
}

#[napi]
impl DatabaseService {
    #[napi]
    pub async fn initialize(&mut self, database_url: String) -> Result<bool> {
        let pool = PgPoolOptions::new()
            .max_connections(10)
            .connect(&database_url)
            .await
            .map_err(|e| napi::Error::new(napi::Status::GenericFailure, e.to_string()))?;
        
        self.pool = Some(pool);
        Ok(true)
    }
}
```

## 📚 Best Practices

### 1. Error Handling

- Use `thiserror` for structured error types
- Implement `From` traits for error conversion
- Provide meaningful error messages
- Include error context and details

### 2. Async Operations

- Use `tokio` for async runtime
- Avoid blocking operations in async contexts
- Use appropriate concurrency patterns
- Handle cancellation gracefully

### 3. Resource Management

- Initialize resources in `initialize()` method
- Clean up resources properly
- Use connection pooling for databases
- Monitor memory usage

### 4. Configuration

- Validate configuration parameters
- Provide sensible defaults
- Support environment variables
- Document all configuration options

### 5. Logging

- Use structured logging with `tracing`
- Include relevant context in log messages
- Use appropriate log levels
- Avoid logging sensitive information
